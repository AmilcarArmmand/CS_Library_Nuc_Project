Problem Statement

The library existed, but the system around it was mostly manual.

Borrowing depended on handwritten records, memory, and follow-up. That worked for a small room, but it made availability hard to trust and made the collection harder to manage over time.

  • Students had no quick way to check what was available before visiting.
  • Admins had to interpret scattered checkout information by hand.
  • The in-person flow needed to work with real student IDs, scanners, and touch hardware.
The original paper log became the clearest proof of the problem: useful information existed, but it was not searchable, enforceable, or easy to keep current.
Architecture Diagram

Three interfaces share one backend and one set of library records.

The design keeps the student website, admin dashboard, and kiosk separate where the user experience needs to differ, while keeping catalog and loan data centralized.

Web Portal Students browse, search, view details, and check account activity.
Kiosk Touchscreen station for student ID entry and scanner-assisted checkout.
Admin Dashboard Catalog editing, user oversight, requests, and loan management.
Express REST API
Application Layer Node.js, Express, TypeScript, EJS, Passport, and shared route logic coordinate authentication, search, checkout, returns, holds, donations, and account workflows.
Data and integrations

Solution

We built the flow around how the library is actually used.

The project is not just a catalog page. It connects discovery, borrowing, returns, and staff management into one system.

1
Browse before walking in The web portal lets students inspect the collection and see availability without needing to be in the room.
2
Check out at the kiosk The kiosk supports student ID login, touch navigation, scanner input, checkout, return, renew, and donation entry points.
3
Manage from one place Admins can maintain records, review activity, and keep the department collection current.

Highlights / Lessons Learned

What we learned while turning it into something usable

  • Trust mattered. Once we trusted each other to own pieces of the system, delegation and debugging became much smoother.
  • Hardware changes the truth. A kiosk that feels fine on a laptop still needs real scanner and touchscreen testing.
  • Deployment exposed issues that local development could not show, especially around routing, authentication, and long-running services.
  • The strongest demo is not the biggest feature list. It is a clean flow that people can understand immediately.
Our biggest takeaway was that software for a real space has to respect the people, hardware, and habits already in that space.
